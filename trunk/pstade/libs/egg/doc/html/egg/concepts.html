Constant Object

Description

A Constant Object is an object which is const qualified.

Refinement of
  • ...
Requirements

For any Constant Object x the following must be met:

  • boost::remove_reference<_decltype(x)>::type is _typeof(x) const.
Example 
int const x = 0; // x is a ConstantObject.

Lexically Typed Object

Description

A Lexically Typed Object is a named lvalue whose type is referable by using prefix T_.

Refinement of
  • ...
Requirements

For any Lexically Typed Object x the following must be met:

  • x is a named lvalue.
  • T_++x is _typeof(x).
Example 
int x;
typedef int T_x; // Now x is a LexicallyTypedObject.

Bindable Object

Description

A Bindable Object is a Function Object which can be passed to boost::lambda::bind.

Refinement of
  • Function Object
Requirements

In addition to the requirements defined in the refinements, for any Bindable Object b the following must be met:

  • There is a N such that boost::lambda::bind(b, boost::lambda::_1,...,boost::lambda::_++N) is a valid expression.

Pipable Object

Description

A Pipable Object is an object which behaves as if it were a "pipe" using operator| and operator|=.

Refinement of
  • Copy Constructible
Requirements

In addition to the requirements defined in the refinements, for any Pipable Object p the following must be met:

  • x|p(x1,...,xN) is a valid expresion if and only if p(x1,...,xN)|=x is a valid expression.
  • x|p(x1,...,xN) and p(x1,...,xN)|=x have the same effects.
  • If N is 0, then x|p(), x|p, p|=x, and p()|=x are valid expressions which have the same effects.

Static Function Object

Description

A Static Function Object is a Function Object which is intended to replace normal functions. Egg is a library which helps you define Static Function Object.

Refinement of
  • Bindable Object
  • Constant Object
  • Deferred Callable Object
  • Lexically Typed Object
Requirements

In addition to the requirements defined in the refinements, for any Static Function Object f the following must be met:

  • _typeof(f) is a POD type, and f is statically initialized.
  • boost::result_of<_typeof(f)(a1,...,aN)>::type is the same as boost::result_of<_typeof(f) const(a1,...,aN)>::type.
Example 
typedef int (*T_foo)(double d);
int foo_impl(double d) { ... }
T_foo const foo = &foo_impl; // foo is a StaticFunctionObject.

Auxiliary Class Interface

Description

An Auxiliary Class Interface is a Static Function Object which can be used as if it were a member function.

Refinement of
  • Static Function Object
Requirements

In addition to the requirements defined in the refinements, for any Auxiliary Class Interface i the following must be met:

  • x|i(x1,...,xN) is a valid expression if and only if i(x, x1,...,xN) is a valid expression.
  • x|i(x1,...xN) and i(x, x1,...,xN) have the same effects.
  • If N is 0, then x|i() and x|i are valid expressions which have the same effects.

result_of tutorial

Caution

Boost.ResultOf document in boost.org is outdated.

How to call

If you pass an lvalue argument, it must be a reference type; It can't be a reference type, otherwise: 

#include <boost/utility/result_of.hpp>
using boost::result_of;

struct T_plus
{
    typedef int result_type;

    result_type operator()(int i, int j) const
    {
        return i + j;
    }
};

void test_lvalue()
{
    int i = 2;
    result_of<T_plus(int &, int)>::type r = T_plus()(i, 3);
    BOOST_CHECK( r == 5 );
}

Thanks to these two convention, you can easily write a "chain" of Boost.ResultOf: 

void test_chain()
{
    int const i = 3;
    result_of<
        T_plus(result_of<T_plus(int, int)>::type, int const &)
    >::type r
        = T_plus()(T_plus()(1, 2), i);
}

A callable type can't be a reference type but must be "const-correct", meaning that you must add const to const-qualified Function Object type: 

T_plus const plus = {};

void test_const()
{
    result_of<T_plus const(int, int)>::type r = plus(2, 3);
    BOOST_CHECK( r == 5 );
}

However, if a library guarantees the same behavior between const-qualified type and non-const-qualified type (as Static Function Object does so), you could omit const: 

void test_operatable()
{
    result_of<T_plus(int, int)>::type r = plus(2, 3);
    BOOST_CHECK( r == 5 );
}

Bugs
  • msvc-7.1 and msvc-8.0 doesn't work with POD templates.
  • A const-qualified function pointer doesn't work.

These bugs can be worked around by pstade::result_of in <pstade/result_of.hpp>. The first bug is fixed with Boost1.35.
