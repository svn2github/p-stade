Function Adaptor is a higher-order function that takes a "base" Function Object and returns an "adapted" Function Object. Function Adaptor provides, if possible, two interfaces: A normal higher-order Function Object, and an MPL Metafunction with the corresponding macro for static initialization. An adapted non-local Function Object with static storage duration is statically initialized if the "base" Function Object type is POD and an expression passed to the macro is a constant expression. Also, Function Adaptors don't modify Default Constructible and Copy Assignable-ness of base Function Object unless otherwise specified.

Important

If a macro argument is not guaranteed to contain no commas and recursions, you must use BOOST_EGG_XXX_L ... BOOST_EGG_XXX_R instead of BOOST_EGG_XXX(...).

fix

Description

fix is "fixed point combinator", which enables you to write recursive functions on the fly.

Header
  • <boost/egg/fix.hpp>
Model of
  • Static Function Object
Valid expressions

Valid expression

Semantics

fix(__pfo)(a1)

__const(__pfo)(fix(__pfo))(__fwd_arg_list([a1], by_perfect))

fix2

Static compose(fix, curry2)

Preconditions
  • The corresponding semantics is a valid expression.
Note

fix(__pfo) that is passed to __pfo as the first argument represents __pfo itself. Also notice that __pfo must be curried in advance.

Example 

void egg_example()
{
    using bll::_1;
    using bll::_2;

    int r =
        fix2(
            bll::ret<int>(
                // \(f,a) -> a == 0 ? 1 : a * f(a-1)
                bll::if_then_else_return( _2 == 0,
                    1,
                    _2 * lazy(_1)(_2 - 1)
                )
            )
        ) (5);

    BOOST_CHECK(r == 5*4*3*2*1);
}

    typedef void result_type;

    int a[] = {1,2,3};

    BOOST_CHECK(c.m_count == 3);

nestN

Description

nestN is generalization of curryN and lazy, which enables you to write nested lambda expressions. N represents nesting level of lambda expression.

Header
  • <boost/egg/nest.hpp>
Model of
  • Static Function Object
Notation
  • bll_bind is an imaginary Function Object behaving as if it were boost::lambda::bind.
  • bind_ is bll_bind if Bind is __ud, Bind() otherwise.
  • protect_ is boost::lambda::protect if Protect is __ud, Protect() otherwise.
  • LAZY(N) is:
    • If N == 0, X_identity<by_value>().
    • Else if N == 1, X_lazy<__typeof(bind_)>().
    • Otherwise, X_lazy<result_of_lazy<__typeof(bind_), result_of_lazy<__typeof(bind_),...N-1 times...>::type>::type>().
Valid expressions

Valid expression

Semantics

X_nest%%N<Bind = __ud>

A Major Function Object type

X_nest%%N<Bind>()(f)(a1,...,aK)

Polymorphic LAZY(N)(f)(__fwd_arg_list(a, by_value))

nest%%N(x)

X_nest%%N<>()(x)

XX_%%M%%_<Protect = __ud>

A Major Function Object type

XX_%%M%%_<Protect>()(p)

Polymorphic protect_(protect_(...M times...(p)...))

_%%M%%_

Static XX_%%M%%_<>()

Preconditions
  • 0 <= N && N <= BOOST_EGG_MAX_ARITY.
  • 0 <= M && M <= N-1.
  • The corresponding semantics is a valid expression.
Example 

void egg_example()
{
    using bll::_1;
    using bll::_2;
    std::plus<int> plus;
    std::minus<int> minus;

    int i6 = 6, i1 = 1, i3 = 3, i9 = 9;

// Lv: 0     1      2          3
    // \x -> (\y -> (\(z,w) -> foo(y,w,z,x))))
    BOOST_CHECK( nest3(foo)(_1_(_1), _2_(_2), _2_(_1), _0_(_1)) 
        (i3)(i6)(i1,i9) == foo(6,9,1,3) );

    // \x -> apply(\y -> minus(x,y), plus(x,3))
    BOOST_CHECK( nest1(apply)(nest2(minus)(_0_(_1), _1_(_1)), nest1(plus)(_0_(_1), 3)) 
        (i9) == minus(9, plus(9,3))  );
}

_M_ turns a placeholder into nested one.

By the definition, nest1 has the same semantics as lazy.

Tip

When you provide a Pipable Function Object, its "base" function too should be provided with consistent naming convention. So that, your clients can use it with result_of.

Example 

struct weird_mult
{
    typedef int result_type;

    int operator()(int x) const
    {
        return x * x;
    }

    int operator()(int x, int y) const
    {
        return x * y;
    }

    int operator()(int x, int y, int z) const
    {
        return x * y * z;
    }
};

result_of_pipable<weird_mult>::type const mult = BOOST_EGG_PIPABLE({});

void egg_example()
{
    BOOST_CHECK( ( 2|mult ) == 2 * 2 );
    BOOST_CHECK( ( 2|mult() ) == 2 * 2 );
    BOOST_CHECK( ( 2|mult(3)|mult(4) ) == 2 * 3 * 4 );
    BOOST_CHECK( ( 2|mult(3, 4) ) == 2 * 3 * 4 );

    BOOST_CHECK( ( mult|=2 ) == 2 * 2 );
    BOOST_CHECK( ( mult()|=2 ) == 2 * 2 );
    BOOST_CHECK( ( mult|=mult|=2) == (2 * 2) * (2 * 2) );

    BOOST_CHECK( ( mult(3)|=2 ) == 2 * 3 );
    BOOST_CHECK( ( mult(3, 4)|= 2 ) == 2 * 3 * 4 );
}

Tip

According to Charles Brockman's research, "pipable" is not misspelling of "pipeable".
